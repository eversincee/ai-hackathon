Автоматизация оцифровки паспортов оборудования

Задача 1 — Хакатон РТУ МИРЭА 2026

FastAPI · Surya OCR · Qwen2.5-VL · Next.js · SQLite · Ollama

Контекст задачи

Что такое паспорт оборудования

Официальный документ по ГОСТ Р 2.610-2019, содержащий технические характеристики, правила эксплуатации, хранения и транспортировки. Обязателен для сертификации, регистрации в Гостехнадзоре и инвентаризации на складах.


Проблема

  • Паспорта поступают в бумажном виде — на стандартных бланках, по форме производителя или в произвольной форме
  • Ручной перенос данных в 1С занимает часы и сопряжён с ошибками
  • Физические документы сложно отслеживать на этапах приёмки, хранения и отгрузки

Цель задачи

Создать комплексную систему перевода паспортов в цифровой формат и автоматизированного управления в едином цикле — от изготовления до поставки заказчику.


Требования организаторов
  • Распознавание текста и цифровых данных с отсканированных паспортов
  • Занесение данных в табличный формат 1С
  • Генерация штрихкодов (Code 39 / Code 128 / EAN-13) для каждого паспорта и шкафа
  • Идентификация оборудования по штрихкоду через сканер (ТСД, лазерный, 2D)
  • Формирование общей таблицы паспортов шкафа
Бонусные требования
  • Интеграция с 1С · Обратная связь оператора · Чек-листы и выгрузка отчётов

Ключевая идея: двойное распознавание

Одного OCR недостаточно — реальные сканы бывают повёрнуты, засвечены, с печатями поверх текста. Запускаем два независимых метода параллельно и сравниваем результаты.

Ветка A — VLM (Qwen2.5-VL)

Нейросеть анализирует изображение страницы целиком и сразу формирует структурированный JSON.


Преимущества:

  • Понимает структуру таблиц и расположение полей
  • Устойчива к низкому качеству скана и наклонам
  • Не зависит от качества OCR-слоя
Ветка B — OCR → LLM (Surya + Qwen2.5)

Surya извлекает текст со скана, Qwen2.5-Instruct структурирует его в JSON по заданной схеме.


Преимущества:

  • Выше точность на чётком печатном тексте
  • Меньше ошибок в числовых значениях и кодах
  • Оптимизирован под русский язык
Оба источника совпали
Уверенность 100% · поле выделено зелёным
Один источник пуст
Уверенность 70% · требует проверки
Источники расходятся
Уверенность 50% · ручная корректировка

Pipeline обработки документа

  1. Загрузка — пользователь загружает PDF или изображение через веб-интерфейс
  2. Предобработка — PyMuPDF разбивает PDF на отдельные страницы, выполняется автоматическая коррекция ориентации
  3. Параллельное распознавание — VLM и OCR→LLM запускаются одновременно, независимо друг от друга
  4. Объединение результатов (merge) — поля сравниваются, каждому присваивается коэффициент уверенности
  5. Валидация — проверка форматов номера документа, серийных номеров, соответствия производителя, наличия даты выпуска
  6. Маршрутизация — уверенность >90% и нет ошибок → автоматическое одобрение; иначе → очередь ручной проверки
  7. Ревью оператора — скан и форма с полями отображаются рядом; цветовая индикация уверенности по каждому полю
  8. Штрихкод и экспорт — генерация Code 128, PDF-этикетки, CSV / CommerceML XML / PDF чек-лист

Почему локальные модели

Конфиденциальность данных

Паспорта оборудования содержат технические характеристики, серийные номера и данные производителей. При использовании локальных моделей документы не покидают инфраструктуру предприятия.


Автономная работа

После первоначальной загрузки моделей система функционирует без подключения к интернету — в том числе в производственных и складских помещениях с ограниченной сетевой доступностью.

Валидация и контроль качества

Распознать данные недостаточно — необходимо убедиться в их корректности.

Проверяемые условия

  • Формат номера документа соответствует шаблону
    БНРД.XXX.XXX ПС, TREI.XXX.XXX ПС
  • Серийные номера соответствуют паттерну производителя
    TREI: A1B2345 · TECON: 7 цифр
  • Наименование производителя соответствует префиксу документа
  • При наличии гарантийного срока указана дата выпуска

Справочник производителей

Файл vendors.yaml содержит список производителей с паттернами номеров документов и серийников. Расширяется без изменения кода.


Данные валидны, уверенность >90% — автоматическое одобрение
Предупреждение — оператор проверяет поле
Ошибка валидации — обязательная ручная корректировка

Интерфейс оператора

Страница проверки (ревью)

  • Скан документа и форма отображаются рядом
  • Каждое поле подсвечено по уровню уверенности
  • Клик по полю — подсветка соответствующей области на скане
  • Кнопки «Одобрить» / «Отклонить»

Прочие разделы интерфейса

  • Загрузка — drag-and-drop, поддержка PDF и изображений
  • Сборки — группировка паспортов шкафа
  • Сканирование — поиск по штрихкоду
  • Экспорт — CSV, XML, PDF чек-лист
  • Импорт 1С — привязка к строке номенклатуры

Сборки, штрихкоды и экспорт

Штрихкоды Code 128
  • Уникальный идентификатор каждого паспорта
  • PDF-этикетка для печати и размещения на оригинале
  • Идентификация через сканер ТСД / лазерный / 2D
  • Поиск карточки по коду в веб-интерфейсе
Сборки (шкафы)
  • Объединение паспортов одного шкафа в единую запись
  • Указание названия, кода проекта, заводского номера
  • Штрихкод на сборку в целом
  • Сводная таблица в формате Приложения 4
Экспорт
  • CSV с BOM для корректного открытия в Excel
  • CommerceML XML — стандарт обмена данными 1С
  • PDF чек-лист для приёмки на складе
  • Привязка к номенклатуре по коду СУ
